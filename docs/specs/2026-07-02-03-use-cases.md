**Status:** Implementado

---

## Objetivo

Implementar toda a lógica de aplicação como use-cases puros: classes `@Injectable()` que dependem apenas de `IProjectsRepository` (via DI), sem nenhuma dependência HTTP ou de infraestrutura.

---

## Princípios

- Cada use-case faz **exatamente uma coisa**
- Nenhuma lógica de negócio no controller — tudo aqui
- `risk` nunca é calculado no use-case — delegado à entidade
- Erros são lançados como exceções (`NotFoundException`, `BadRequestException`) para o filter global tratar

---

## Use-Cases Implementados

### `CreateProjectUseCase`
- Recebe `ProjectProps` (campos em inglês)
- Chama `Project.create()` para instanciar com risco calculado e status `analysis`
- Persiste via `repository.create()`
- Retorna o `Project` criado

### `ListProjectsUseCase`
- Retorna todos os projetos via `repository.findAll()`

### `GetProjectUseCase`
- Busca por ID; lança `NotFoundException` se não encontrado

### `UpdateProjectUseCase`
- Busca o projeto, chama `project.update(props)` (que recalcula risco internamente se orçamento/datas mudarem)
- Persiste via `repository.update()`

### `DeleteProjectUseCase`
- Busca o projeto; lança `BadRequestException` se `!project.isDeletable()` (status `in_progress` ou `closed`)
- Remove via `repository.delete()`

### `UpdateProjectStatusUseCase`
- Busca o projeto; chama `project.transitionTo(newStatus)`
- A entidade valida a transição — o use-case não valida
- Persiste via `repository.update()`

### `GetAiAnalysisUseCase`
- Busca o projeto; monta `AiAnalysisInput` com os dados do projeto
- Delega para `AiAnalysisService.analyze()` e retorna o resultado

---

## Testes

Cada use-case tem seu próprio arquivo de spec (`*.use-case.spec.ts`) usando `InMemoryProjectsRepository` como repositório real (sem mocks). Cobre:
- Caminho feliz
- Projeto não encontrado
- Regras de negócio (deleção, transições inválidas)
